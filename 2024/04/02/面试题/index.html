什么是RPC框架? 优点？

RPC框架：实现分布式系统中跨网络进行通信的技术，是一种计算机通信协议。

作用：调用服务提供方提供的方法时，像在本地调用自己的方法一样调用他人写的方法。

优点：

1.自主选择通信协议：在我的RPC框架可以提供了两种通信协议的选择，包括HTTP和TCP协议，其中TCP协议效率更高，因为TCP协议是在传输层上的，与HTTP协议比，少了HTTP请求头，传输的效率更高，性能也更高

2.拥有额外的功能

并且在我做的RPC框架中增加了额外的功能：

包括：

​ 1.负载均衡：我编写了负载均衡器通用的接口，并提供了轮询（使用了轮询负载均衡器）、随机（Java自带的Random类实现）、一致性Hash的方式。根据主流框架Dubbo文档我设计了通过读取配置的方式选择对应的负载方式。

​ 2.服务发现：服务提供者在注册中心注册服务，而消费者则根据反射获取到服务提供者对应的方法。

​ 3.容错机制: 在我的项目中想到了两种容错设计方案，并将这两种设计方案结合来针对不同的场景使用不同的容错策略。

​ 1.当出现系统错误时，该错误是由网络波动、服务端临时不可用导致的，可以进行重试操作。重试多次仍然失败，可能产生了更严重的问题，触发其他容错机制。

​ 我在这个项目中设计的容错机制包括快速失败和静默处理

​ 快速失败：立刻通知外层调用方失败

​ 静默处理: 遇到异常后，记录一条日志，然后正常返回一个响应对象，就好像没有出错。

​ 4.重试机制:我设计了两种重试方案，固定重试间隔和不重试，并引入Guava-Retrying重试库取实现

不设计重试方案行吗？并且为什么要用这个库？

不行，因为我们的后端业务系统可能会出现接口调用失败、网络拥塞超时、任务执行失败、系统错误等异常情况，需要进行重试操作。但某些场景下我们对重试有特殊要求，比如延迟重试、降频重试等，此时自己编写重试代码会很繁琐，在 Java 中，可以使用 guava-retrying 帮我们实现灵活的重试机制。

为什么采用这种技术，优势是什么?

1.guava-retrying 是一个线程安全的 Java 重试类库，提供了一种通用方法去处理任意需要重试的代码，可以方便灵活地控制重试次数、重试时机、重试频率、停止时机等，并具有异常处理功能。

2.相比我们自己实现重试机制，使用guava-retrying有如下优势

​ 1. 对代码的侵入性更小

​ 2.更直观，改动方便

​ 3.可复用重试器至多个任务（代码段）

3.从底层源码来看，它能够应用多个场景

​ 1.重试的条件

​ 根据执行结果判断是否重试 retryIfResult

​ 发生异常时重试

​ 2.等待策略

​ FixedWaitStrategy：固定等待时长策略，比如每次重试等待5s

​ RandomWaitStrategy：随机等待时长策略，每次重试等待指定区间的随机时长

​ IncrementingWaitStrategy：递增等待时长策略，指定初始等待值，然后重试间隔随次数等差递增，比如依次等待10s、30s、60s（递增值为10）

​ ExponentialWaitStrategy：指数等待时长策略，指定初始值，然后每次重试间隔乘2（即间隔为2的幂次方），如依次等待 2s、6s、14s。可以设置最大等待时长，达到最大值后每次重试将等待最大时长。

​ FibonacciWaitStrategy ：斐波那契等待时长策略，类似指数等待时长策略，间隔时长为斐波那契数列。

​ CompositeWaitStrategy ：复合时长等待策略，可以组合多个等待策略，基本可以满足所有等待时长的需求

​ 3.停止策略：停止策略决定了何时停止重试，比如限制次数、时间等，通过 withStopStrategy 方法注册等待策略：

​ 4.超时限制：通过 withAttemptTimeLimiter 方法为任务添加单次执行时间限制，超时则中断执行，继续重试。

​ 5.监听器：可以通过 withRetryListener 方法为重试器注册***，每次重试结束后，会按注册顺序依次回调 Listener 的 onRetry 方法，可在其中获取到当前执行的信息，比如重试次数等。

RPC与HTTP的区别：

相同点：都是应用层协议

设计目的：RPC是为了方便调用方调用服务提供者提供的方法，而HTTP是客户端和服务器之间传输超文本文档，用于Web应用程序之间的通信

实现方式：RPC可以自定义协议和编码机制，可扩展性更强（在我的项目中我是使用TCP加自定义协议头实现消息传输，可以提供更高的性能），而HTTP只基于TCP协议，通过请求-响应模型进行通信，采用文本格式的消息头和消息体。

使用场景：RPC更多运用与服务端之间的通信，而HTTP可同时运用于服务器端、后端和前端通信

Dubbo（阿里巴巴）、Feign（Netflix基于HTTP）。

首先我必须承认，我的RPC项目无法跟完善的RPC项目比，但是我在设计实现自己的RPC项目时，借鉴了Dubbo的设计理念

我的RPC框架有如下的特点

1.自定义RPC协议：我的RPC项目使用了Vert.x设计实现了TCP服务器和客户端，并自主设计了消息头的格式，相比Feign这种HTTP的传输性能更高

2.可扩展性：我的RPC项目大量运用SPI机制，扩展性强，使用者可以使用SPI实现自定义接口，项目的基本上所有模块都支持自定义

3.自实现注册中心：使用Etcd分布式云原生存储中间件实现注册中心，Etcd是用Go语言开发，分布式键值存储系统，具有高性能，强一致的特点。

4.开箱即用：使用基于注解和Spring Boot Starter的项目启动机制。

系统的核心架构设计？有哪些模块？作用？关系？

1.消费者代理模块：基于代理模式实现，服务消费者无需考虑是如何实现调用的，直接调用写好的服务接口即可.

2.注册中心: 服务提供者会将自己的基本信息（包括自己的地址、接口、分组等详细信息）报到注册中心模块，并且将服务上线、下线的时候通知到注册中心，服务调用方就可以从注册中心动态获取调用信息。

3.路由模块：当有多个服务和服务提供者节点时，通过实现不同算法的负载均衡器，帮助服务消费者选择服务节点发起调用。

4.本地服务注册器：服务提供端存储服务名称和实现类的映射关系，便于根据消费者发送的服务名获取对应的实现类，从而通过反射完成调用

5.自定义RPC协议：我在这个项目中实现了基于Vert.x(TCP)和自定义请求头的网络传输协议，并自主实现了对字节数组的编码和解码器，从而可以更高性能完成请求和响应

6.请求处理器：服务提供者接收到请求后，解析接收的请求，在本地服务注册器中找到服务实现类并调用方法

7.重试和容错模块:保证服务的可用性和稳定性

8.启动器模块：用注解驱动+Spring Boot Starter,开发者可以快速引入RPC框架到项目中.
